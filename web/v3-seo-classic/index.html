    1. The problem: Lovable apps are React SPAs

    When you build a landing page in Lovable, the tool generates a Vite + React + TypeScript application. The published version of your page is a single-page application (SPA). When a user opens it in a browser, JavaScript runs, React renders your content, and everything looks fine.

    When a search engine crawler or an LLM's web fetcher requests your page, the response is essentially this:

    <!doctype html>
<html>
  <head><title>Your App</title></head>
  <body>
    <div id="root"></div>
    <script type="module" src="/assets/index-abc123.js"></script>
  </body>
</html>

    That <div id="root"> is empty. Your hero copy, features, pricing, FAQ — none of it is in the HTML. It only exists after JavaScript executes.

    Google can execute JavaScript to render some pages, but it does so on a delayed, deprioritized "second-pass" basis, and inconsistently. ChatGPT, Perplexity, and most LLM web fetchers don't execute JavaScript at all. They see your page as a blank document. So they don't cite you.

    Quick check: open your Lovable app's live URL, view-source, and search for any of your hero copy. If it's not in the source, your page is invisible to most of the AI-search world.

    2. Why this matters more in 2026

    Two trends turn this from "minor SEO oversight" into a critical distribution problem:

    1. Search is migrating from Google to LLMs. When a buyer asks ChatGPT "what's the best receipt scanner for freelancers?", the answer comes from URLs the model can read. If your page is invisible, you don't exist in that answer.
    2. Lovable, Bolt, v0, and Base44 made it cheap to ship landing pages. Most of them ship as SPAs by default. Most founders don't realize what that means for distribution. The pile of invisible-but-beautiful pages is growing fast.

    The fix is mature SEO advice (server-side render, or prerender to static HTML) plus newer GEO practices (extractable answers, comparison tables, JSON-LD schema). Most existing SEO tools handle the first; few handle the second; almost none handle both for a Lovable repo specifically.

    3. The 5-step pipeline

    The lovabletoseo CLI takes your GitHub repo URL and runs the following stages. Each stage produces a JSON or HTML artifact you can inspect — there is no black box.

    Step 1 — Ingest

    We git clone your Lovable repo, detect that it's a Vite + React stack, and walk the source tree. We need read access to the repo, not write — the output goes on a new branch you control.

    Step 2 — Prerender

    An LLM reads your index.html, App.tsx, and every component under src/, then produces a single static HTML document with all the page content baked in. Same Tailwind classes, same hydration scripts — meaning crawlers see your full content immediately, and the page still works as an interactive React app for users.

    Step 3 — Diagnose

    We hit three Peec AI REST endpoints in parallel:

    • POST /reports/brands — your visibility, share-of-voice, sentiment, and average position vs competitors across LLMs
    • POST /queries/search — the actual queries buyers ask LLMs that touch your category
    • POST /reports/urls — which URLs LLMs cite when they answer those queries

    Step 4 — Strategize and enhance

    Two LLM calls: a strategist writes a markdown brief grounded in the Peec signal, and an enhancer applies that brief to the prerendered HTML. The enhancer's job is surgical — it adds FAQ blocks, comparison tables, JSON-LD, and tighter meta tags, but it does not redesign the page or change the brand voice.

    Step 5 — Ship

    The final static site (index.html + robots.txt + sitemap.xml) is committed to seo/ on a new branch. With --open-pr we run gh pr create. You review the diff and merge if you like it.

    5. What "GEO" actually means

    Generative Engine Optimization is the practice of structuring a page so LLMs cite it when they answer questions. The high-leverage moves, in order:

    Direct-answer-first
    The first sentence under each H2 is a complete, self-contained answer to the implicit question. LLMs lift the first 1-2 sentences as their quote.
    Comparison tables
    For every "X vs Y" buyer query, ship a table with concrete rows. LLMs lift table rows wholesale. Include competitors honestly — pages that flatter only themselves don't get cited.
    Cited statistics
    Every numeric claim links to a source. Unsourced confidence gets discarded as marketing language. If you can't cite it, don't claim it.
    JSON-LD schema markup
    Inline <script type="application/ld+json"> blocks for Organization, SoftwareApplication (or Product), and FAQPage. This is how engines parse your entities cleanly.
    Entity consistency
    The first mention defines the entity (your brand name + one-line descriptor). Every later mention uses the same surface form. Drift hurts retrieval.
    Extractable lists
    Prefer <ul> and <ol> over prose for feature/benefit content. Models lift lists cleanly; prose gets paraphrased and the specifics smoothed away.

    Notice that this very page applies all six. That's deliberate. If a "GEO tool" can't make its own landing page rank, it has no business selling you the practice.

    6. FAQ

    Why isn't my Lovable app showing up on Google?
    Lovable apps are Vite + React SPAs. The HTML served to crawlers is essentially an empty <div id="root">. Crawlers don't execute JavaScript reliably, so Google indexes nothing. The fix is server-side or static rendering.
    What is GEO?
    Generative Engine Optimization. The practice of structuring your website so LLMs (ChatGPT, Claude, Perplexity, Gemini) cite it when they answer buyer questions. Overlaps with SEO but emphasizes extractable answers, comparison tables, citable stats, and schema markup.
    Will lovabletoseo break my Lovable app?
    No. We open a pull request that adds files under seo/ on a new branch. Your existing React source is untouched. Review, merge, close, or ignore — it's your repo.
    How is this different from LovableHTML?
    LovableHTML is a transparent prerender proxy at the DNS layer — it renders whatever content is on your page, without changing it. lovabletoseo also rewrites the content based on Peec data: which queries buyers actually ask LLMs, which URLs LLMs cite, and where competitors out-rank you. Indexable vs cited is the gap we close.
    Do I have to migrate off Lovable?
    No. Lovable stays the editor. We deploy the static site on a different subdomain (typically www.yourbrand.com) while the editable app stays at yourbrand.lovable.app.
